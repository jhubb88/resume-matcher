## 1. TL;DR

The AI Resume Matcher compares a candidate resume against a job description and returns a structured match report — score, hiring tier, skill gaps, and coaching notes — generated by Claude. The notable parts are the async self-invoke Lambda pattern (the user never holds a 30-second connection; the frontend submits, gets a job ID, then polls), and the reproducible CI build that rebuilds the 1,961-file Lambda package from a pinned `requirements.txt` on every backend-changing commit. Stack: vanilla HTML/JS frontend on CloudFront + S3, Python 3.11 Lambda behind API Gateway, Anthropic SDK calling `claude-sonnet-4-6` directly.

## 3. Architecture

The frontend is a single-page `index.html` served from S3 through CloudFront with custom domain `resume-matcher.jimmyhubbard2.cc`. The page renders an upload UI with two drop zones (resume + job description), collects the candidate name, and posts `{name, resume_base64, resume_type, job_description}` to `POST /analyze` on API Gateway. API Gateway requires `x-api-key`, rate-limits via usage plan `vboqy8` (2 rps, 5 burst, 100 per day), and proxies to the Lambda handler.

The Lambda handler writes a `status.json` to the results bucket and immediately self-invokes asynchronously with the same payload plus a 12-character hex job ID. The sync handler returns `{"job_id": "..."}` to the user within milliseconds. The async invocation extracts text from the resume (decodes base64; runs `pypdf.PdfReader` for PDFs, UTF-8 decode for plain text), assembles the analysis prompt, calls the Anthropic API directly using the `anthropic` SDK with `claude-sonnet-4-6`, parses the returned JSON, and writes `{job_id}/result.json` to the results bucket. The async pattern keeps the API Gateway 29-second integration timeout out of play; Claude calls take 5-15 seconds and any of that latency would put the user-facing 200 at risk.

The frontend polls `GET /result/{jobId}` every few seconds. The handler reads from the results bucket: returns 200 with the rendered report once `result.json` exists, 202 with `{"status": "processing"}` while only `status.json` is present. The frontend renders the report once 200 returns. Result objects auto-expire after 24 hours via S3 lifecycle; the bucket holds PII (the resume text) and should not retain indefinitely.

## 4. Current state

### Works end-to-end

- Upload resume (`.txt` or `.pdf`) and job description through the live URL, get rendered match report back. Verified live 2026-05-12.
- CI auto-deploys on every push to `main`. S3 sync runs unconditionally; Lambda rebuild gated by a diff-guard that checks if `lambda_function.py` or `requirements.txt` changed in `HEAD^..HEAD`.
- CORS preflight against `POST /analyze` returns `Access-Control-Allow-Headers: Content-Type,x-api-key,Authorization`. The CI workflow enforces this; any deploy that produces a Lambda response without `x-api-key` in that header fails the build.
- Async self-invoke pattern keeps the user-facing 200 fast. POST returns `{"job_id": "..."}` in milliseconds; the Claude call runs in a separate async invocation.
- Results bucket auto-expires objects after 24 hours. PII (resume text + analysis) is not retained.

### Open items

- Frontend calls the raw `execute-api.amazonaws.com` URL directly. No CloudFront proxy path exists for `/api/*`. Tracked as Future Enhancement E1.
- No CloudWatch alarms exist for this project. Tracked as E2.
- `portfolio-user` CLI principal lacks `cloudwatch:GetMetricStatistics` and `ce:GetCostAndUsage` permissions. 30-day invocation/error counts and per-project cost are not queryable from the CLI. Live observability is via the AWS Console only. Tracked as E3.
- `deploy.sh` referenced in the README no longer exists in the repo (removed in commit `864ed79`). README is stale on this point.

## 5. Build history

### Phase 1 — Initial scaffold (March 2026)

First end-to-end working version. `deploy.sh` provisioned the IAM role, Lambda function, API Gateway, both S3 buckets, and uploaded the frontend. Single-file Lambda upload at this point (`anthropic` and `pypdf` not yet bundled). Anthropic SDK chosen over Bedrock for the AI call; see Key Decision 1.

### Phase 2 — OAC migration and Public Access Block (2026-05-07)

Frontend bucket `jimmy-resume-matcher` migrated from S3 website endpoint to Origin Access Control. Bucket policy scoped to the `cloudfront.amazonaws.com` service principal with `SourceArn` condition pinned to distribution `E3H0XAJDR3BQG1`. All four Public Access Block settings turned on. Static website hosting disabled. CloudFront `CustomErrorResponses` configured to map 403 and 404 to `/index.html` with a 200 response, giving SPA-style routing without exposing the bucket. Matches the portfolio-wide OAC migration completed across all eight project distributions.

### Phase 3 — Subdomain wiring (2026-05-07)

`resume-matcher.jimmyhubbard2.cc` mapped to distribution `E3H0XAJDR3BQG1` via cPanel CNAME record. ACM certificate provisioned in us-east-1 with DNS validation. CloudFront alias added, viewer certificate set to the ACM ARN, minimum protocol version `TLSv1.2_2021`. cPanel CNAME took roughly 30 minutes to propagate to Namecheap's authoritative nameservers, a known replication-stall pattern resolved by a LiveChat zone-sync request.

### Phase 4 — CORS fix (2026-05-12, commit 55954d0)

The original Lambda CORS configuration was missing `x-api-key` from `Access-Control-Allow-Headers`. Same bug class as Log Analyzer's prior CORS incident: any API Gateway method with `apiKeyRequired=true` requires the preflight to include `x-api-key` in the allowed headers, or browsers block the POST entirely. Fixed by updating the `CORS_HEADERS` constant in `lambda_function.py` to include `Content-Type,x-api-key,Authorization`. Follow-up commit `14f77a7` added a CORS smoke test to the GitHub Actions workflow; every Lambda deploy now runs an OPTIONS request against `/analyze` and exits non-zero if `Access-Control-Allow-Headers` does not contain `x-api-key`. Permanent guard against the same bug returning.

### Phase 5 — CI/CD overhaul (2026-05-12, commits 652c426 + ebd2cee)

Lambda rebuild moved into CI. Pre-overhaul, the Lambda package was assembled manually and uploaded via console or one-off CLI calls — fragile, and on one occasion a manual `zip -j lambda_function.py` call overwrote the entire bundled-dependency package (1,961 files reduced to 1), breaking production until a multi-file rebuild restored it. The new workflow runs `pip install -r requirements.txt -t build/` to install all 17 pinned dependencies into a fresh `build/` directory, copies `lambda_function.py` in, zips the directory contents, and calls `aws lambda update-function-code` with the zip. A sanity check between zip and deploy verifies the package contains more than 100 files; if the count drops below that threshold, the workflow exits with a clear error and the deploy aborts. `wait function-updated` runs after the code update so the next step (CORS smoke test) sees the new code. All GitHub Actions bumped to v6 majors (Node 24).

### Phase 6 — TLS audit (2026-05-14)

Audited reported TLS 1.0 on the REST API, traced it to AWS-managed edge enforcement, confirmed live floor is TLS 1.2+. No action required; raw execute-api closure tracked as E1.

## 7. Key decisions

### 1. Anthropic SDK direct, not Bedrock

The Lambda imports `anthropic` and calls `client.messages.create(model="claude-sonnet-4-6", ...)` directly, with the API key loaded from environment variable `ANTHROPIC_API_KEY`. Bedrock was the alternative path: same model, same prompt, accessed through `boto3.client("bedrock-runtime").invoke_model(...)`. Bedrock would have required a model access request, an additional IAM policy attachment on the Lambda role, and a different SDK shape. The direct path was simpler to ship and gave faster iteration during prompt tuning; no AWS-side model-access waiting and no IAM churn. Trade-off: bound to one provider, no AWS-native logging or audit on the AI call. The Lambda's CloudWatch logs capture the SDK call and response separately, which has been enough so far.

### 2. pip-install-in-CI vs. committing dependencies to git

Every commit that touches `lambda_function.py` or `requirements.txt` triggers a full Lambda rebuild inside GitHub Actions. The workflow runs `pip install -r requirements.txt -t build/` to install all 17 pinned dependencies into a fresh build directory, copies `lambda_function.py` in, zips the contents, and deploys via `update-function-code`. The alternative was committing the `build/` tree (about 7 MB across 1,961 files) into git alongside the source; every dependency upgrade would have been a multi-megabyte diff. CI builds keep the diff small (just the source files plus a one-line `requirements.txt` bump) and put `requirements.txt` in charge as the single source of truth. The CI build takes about 90 seconds for the pip install plus a few seconds to zip and upload, which is acceptable. A `>100 files` sanity check between zip and deploy catches the specific failure mode where a manual single-file zip overwrote the bundled package and broke production for a few minutes; that incident is what triggered the CI overhaul in the first place.

### 3. Async self-invoke pattern (Lambda invokes itself)

The handler routes by `httpMethod` and `job_id` presence. For `POST /analyze`, the sync entry writes a `status.json` placeholder to the results bucket, kicks off `lambda_client.invoke(FunctionName=context.function_name, InvocationType="Event", Payload={...job details with job_id})`, and returns `{"job_id": "..."}` to the user. The async invocation re-enters the same handler with `httpMethod` absent and `job_id` present, runs the PDF text extraction, calls Claude, and writes `{job_id}/result.json`. The frontend polls `GET /result/{jobId}` every few seconds; the handler returns 200 with the rendered report once `result.json` exists, 202 with `{"status": "processing"}` while only `status.json` is present. The alternative was a synchronous flow where the user waits on the original POST connection for 5-15 seconds while Claude responds. API Gateway integration timeout is 29 seconds; a busy or slow Anthropic call would risk timing out, and even a fast call leaves the user staring at a spinning tab for the full call duration. The async pattern decouples the user-facing 200 from the Claude latency entirely. Cost: one extra Lambda invocation per analysis and an S3 round-trip per poll. Both are negligible at portfolio scale.

### 4. OAC over OAI for the static-site bucket

The frontend bucket is fronted by Origin Access Control rather than the older Origin Access Identity pattern. Bucket policy is scoped to the `cloudfront.amazonaws.com` service principal with a `SourceArn` condition pinned to distribution `E3H0XAJDR3BQG1`; CloudFront can read, nothing else can. All four Public Access Block settings are on. Static website hosting is disabled (the bucket uses the S3 REST endpoint via OAC). SPA-style routing is handled at CloudFront with `CustomErrorResponses` mapping 403 and 404 to `/index.html` with a 200 response. OAC is the AWS-recommended pattern since 2022: SigV4 signing, support for SSE-KMS bucket encryption (not used here but available), and a more auditable trust model than OAI's anonymous-reader-by-CF-only convention. Migrated 2026-05-07 as part of the portfolio-wide OAC standardization across all eight project distributions.

### 5. TLS hardening — accepted AWS-managed edge enforcement (2026-05-14)

The `securityPolicy: TLS_1_0` reported by `get-rest-api` looked alarming at first read. EDGE-optimized REST APIs without a custom domain route through an AWS-managed CloudFront layer, and the `securityPolicy` field on the REST API itself is informational unless bound to a custom domain name. No custom domain exists for this API (`aws apigateway get-domain-names` returned `[]`), so the field is vestigial and the live TLS floor for traffic to `https://1kj7anef1b.execute-api.us-east-1.amazonaws.com` is set by AWS at TLS 1.2+. No action taken at the API Gateway layer. The meaningful follow-up is closing the raw execute-api endpoint and forcing all traffic through CloudFront; tracked as Future Enhancement E1.

## 8. Future enhancements

### E1 — Close the raw execute-api endpoint (priority: medium)

The frontend currently posts directly to `https://1kj7anef1b.execute-api.us-east-1.amazonaws.com/prod/{analyze,result/...}`. This works but bypasses CloudFront entirely: no edge caching for the polling GETs, no WAF attachment point, no single TLS chokepoint. Two paths forward:

- **Path A — CloudFront `/api/*` behavior.** Add a second origin to distribution `E3H0XAJDR3BQG1` pointing at `1kj7anef1b.execute-api.us-east-1.amazonaws.com`. Add a cache behavior with path pattern `/api/*` targeting the new origin, `AllowedMethods` set to include POST + GET + OPTIONS, no caching of POST. Rewrite `API_BASE` in `index.html` line 640 from the raw URL to `'/api'`. Update the CORS smoke test in `.github/workflows/deploy.yml` line 85 to hit the CloudFront-proxied path.
- **Path B — API Gateway custom domain.** Provision `api.resume-matcher.jimmyhubbard2.cc` as a custom domain, attach ACM cert, map to API `1kj7anef1b` stage `prod`. Set TLS policy `TLS_1_2`. Rewrite `API_BASE` in `index.html` line 640 to the new custom domain.

Either path eliminates the raw endpoint as a network reach surface. After either lands, set `disableExecuteApiEndpoint=true` and redeploy the prod stage.

### E2 — CloudWatch alarms for errors and 5xx (priority: medium)

Zero alarms exist for this project right now. Suggested baseline:

- Lambda `Errors >= 3 in 5 min` on the `jimmy-resume-matcher` function (catches sustained backend failures: Anthropic outage, S3 write failure, panic in the handler).
- API Gateway `5XXError >= 5 in 5 min` on the `1kj7anef1b/prod` stage (catches integration failures and Lambda invocation errors that surface as 502/504 at the API).
- Anthropic API failures show up as Lambda errors (the SDK raises), so the Lambda alarm covers that path.

SNS topic with email subscription for notifications. One-time setup, low ongoing maintenance.

### E3 — Grant `portfolio-user` CLI observability reads (priority: low)

`cloudwatch:GetMetricStatistics` and `ce:GetCostAndUsage` both return AccessDenied for the `portfolio-user` IAM principal. Today this means scripted observability reads — invocation counts, error rates, per-project cost — fall back to the AWS Console. Acceptable for now (no daily ops burden), but blocks any future scripted reporting. Grant requires either inline policy addition or attached managed policies (`CloudWatchReadOnlyAccess` plus a Cost Explorer policy).

### E4 — Streaming response for incremental render (priority: low)

The async worker today collects the full Claude JSON, writes `result.json` once, then the frontend renders everything at once after the poll succeeds. The Claude call takes 5-15 seconds. The user sees a loading state for that entire window, then everything appears. A streaming pattern would write partial results to S3 as Claude streams them (the Anthropic SDK supports streaming responses); the frontend's existing poll loop already reads from S3 and would naturally pick up partial states. The first `strengths[0]` rendering at 2 seconds feels much better than blank-then-everything at 10 seconds. Architectural change is small: switch the prompt call to `stream=True`, write periodic S3 snapshots from the async worker, render progressively on the frontend as the report grows. Visible UX gain even though end-to-end latency is unchanged.

### E5 — End-to-end automated test in CI (priority: medium)

CI today verifies CORS preflight but not the full POST → poll → result loop. A synthetic resume + JD pair, a real POST, polling for up to 30 seconds, and a check on a known-stable response field would catch regressions in Lambda integration, Anthropic SDK behavior, and PDF parsing; all silent failure modes that the CORS smoke test cannot reach. Cost: one Claude API call per CI run, which at portfolio commit frequency is negligible.

### E6 — Move API key out of client-side code (priority: medium)

Currently visible in index.html via view-source. The 100/day usage plan quota limits damage. Proper fix: route through CloudFront function or Lambda@Edge that injects the key server-side, OR drop API key auth in favor of rate limiting + WAF.
